# Repair a Manually Created Test FMS Account

Use this when a test account was changed to `user_profiles.primary_role = fms` but `/fms/login` still shows:

> Access denied. This account does not have the required ChinaPak ImportHub role.

Changing `user_profiles.primary_role` alone is not enough. FMS access and admin assignment depend on `role_assignments` and `fms_profiles`.

## What `/fms/login` Checks

The FMS login page uses `RoleLoginForm` with `allowedRoles={[USER_ROLES.fms]}`.

After Supabase email/password login succeeds, the server action checks:

1. the Supabase auth session
2. `user_profiles` by `auth_user_id`
3. active rows in `role_assignments`
4. whether one active role is `fms`

`user_profiles.primary_role` is not used as the access grant.

## What `/admin/fms` Reads

The admin FMS directory reads:

1. all rows from `fms_profiles`
2. matching rows from `user_profiles`
3. active `role_assignments` where `role = fms`
4. assignment counts from `fms_assignments`

An FMS is assignable only when:

- `fms_profiles.status = active`
- the linked `user_profiles.id` has an active `role_assignments` row with `role = fms`

If there is no `fms_profiles` row, the account will not appear in `/admin/fms`.

## Required Columns

### `user_profiles`

Important columns:

- `id`
- `auth_user_id`
- `display_name`
- `primary_role`
- `preferred_language`
- `status`
- `metadata`

For a test FMS, use:

- `primary_role = fms`
- `preferred_language = zh` or `en`
- `status = active`

### `role_assignments`

Important columns:

- `id`
- `user_profile_id`
- `role`
- `status`
- `assigned_at`
- `revoked_at`
- `metadata`

Required FMS access row:

- `user_profile_id = user_profiles.id`
- `role = fms`
- `status = active`

### `fms_profiles`

Important columns:

- `id`
- `user_profile_id`
- `fms_code`
- `tier`
- `city_province`
- `categories`
- `academy_status`
- `quality_score`
- `status`
- `metadata`

Required FMS directory row:

- `user_profile_id = user_profiles.id`
- `fms_code` unique
- `tier = bronze`, `silver`, or `gold`
- `status = active`

Safe test values:

- `fms_code`: generated by the SQL snippet as `FMS-TEST-` plus the first 8 auth UUID characters
- `tier`: `silver`
- `status`: `active`
- `quality_score`: `85`
- `city_province`: `Guangzhou, Guangdong`
- `categories`: `electronics`, `bags`, `household goods`
- `metadata.languages`: `["zh-CN","en"]`

## Step 1: Find the Account by Email

Replace `test-fms@example.com` with the test account email.

```sql
select
  u.id as auth_user_id,
  u.email,
  up.id as user_profile_id,
  up.display_name,
  up.primary_role,
  up.status as user_profile_status
from auth.users u
left join public.user_profiles up on up.auth_user_id = u.id
where lower(u.email) = lower('test-fms@example.com');
```

If this returns no rows, the Supabase Auth user does not exist for that email.

## Step 2: Repair the FMS Role and Profile

This snippet is idempotent for the same email. It creates or updates the `user_profiles` row, reactivates or creates one `fms` role assignment, and creates or updates the linked `fms_profiles` row.

Replace only the email and optional display/location/category values. Do not paste service-role keys or secrets into SQL.

```sql
with params as (
  select
    lower('test-fms@example.com')::text as email,
    'Test FMS Guangzhou'::text as display_name,
    'zh'::text as preferred_language,
    'silver'::public.fms_tier as tier,
    'Guangzhou, Guangdong'::text as city_province,
    array['electronics', 'bags', 'household goods']::text[] as categories,
    'in_progress'::public.training_status as academy_status,
    85::numeric(5,2) as quality_score,
    '{"languages":["zh-CN","en"],"repair_source":"manual_test_fms_account"}'::jsonb as metadata
),
auth_match as (
  select
    u.id as auth_user_id,
    u.email
  from auth.users u
  join params p on lower(u.email) = p.email
),
profile_upsert as (
  insert into public.user_profiles (
    auth_user_id,
    display_name,
    primary_role,
    preferred_language,
    status,
    metadata
  )
  select
    a.auth_user_id,
    p.display_name,
    'fms'::public.user_role,
    p.preferred_language,
    'active'::public.profile_status,
    p.metadata
  from auth_match a
  cross join params p
  on conflict (auth_user_id) do update
    set
      display_name = coalesce(nullif(public.user_profiles.display_name, ''), excluded.display_name),
      primary_role = 'fms'::public.user_role,
      preferred_language = excluded.preferred_language,
      status = 'active'::public.profile_status,
      metadata = public.user_profiles.metadata || excluded.metadata,
      updated_at = now()
  returning id, auth_user_id
),
existing_fms_role as (
  select ra.id
  from public.role_assignments ra
  join profile_upsert pu on pu.id = ra.user_profile_id
  where ra.role = 'fms'
  order by
    (ra.status = 'active') desc,
    ra.created_at desc
  limit 1
),
role_reactivated as (
  update public.role_assignments ra
    set
      status = 'active'::public.role_assignment_status,
      revoked_at = null,
      metadata = ra.metadata || '{"repair_source":"manual_test_fms_account"}'::jsonb,
      updated_at = now()
  from existing_fms_role er
  where ra.id = er.id
  returning ra.id
),
role_inserted as (
  insert into public.role_assignments (
    user_profile_id,
    role,
    status,
    metadata
  )
  select
    pu.id,
    'fms'::public.user_role,
    'active'::public.role_assignment_status,
    '{"assignment_source":"manual_test_fms_account"}'::jsonb
  from profile_upsert pu
  where not exists (select 1 from existing_fms_role)
  returning id
),
role_ready as (
  select id from role_reactivated
  union all
  select id from role_inserted
),
fms_profile_upsert as (
  insert into public.fms_profiles (
    user_profile_id,
    fms_code,
    tier,
    city_province,
    categories,
    academy_status,
    quality_score,
    status,
    metadata
  )
  select
    pu.id,
    concat('FMS-TEST-', upper(left(replace(pu.auth_user_id::text, '-', ''), 8))),
    p.tier,
    p.city_province,
    p.categories,
    p.academy_status,
    p.quality_score,
    'active'::public.profile_status,
    p.metadata
  from profile_upsert pu
  cross join params p
  cross join role_ready rr
  on conflict (user_profile_id) do update
    set
      tier = excluded.tier,
      city_province = excluded.city_province,
      categories = excluded.categories,
      academy_status = excluded.academy_status,
      quality_score = excluded.quality_score,
      status = 'active'::public.profile_status,
      metadata = public.fms_profiles.metadata || excluded.metadata,
      updated_at = now()
  returning id, user_profile_id, fms_code, tier, status
)
select
  am.email,
  pu.id as user_profile_id,
  fp.id as fms_profile_id,
  fp.fms_code,
  fp.tier,
  fp.status as fms_profile_status
from auth_match am
join profile_upsert pu on pu.auth_user_id = am.auth_user_id
join fms_profile_upsert fp on fp.user_profile_id = pu.id;
```

If the final `select` returns zero rows, the email did not match a row in `auth.users`.

## Step 3: Verify the Repair

Replace the email and run:

```sql
select
  u.email,
  up.id as user_profile_id,
  up.display_name,
  up.primary_role,
  up.status as user_profile_status,
  ra.id as fms_role_assignment_id,
  ra.role as assigned_role,
  ra.status as role_assignment_status,
  fp.id as fms_profile_id,
  fp.fms_code,
  fp.tier,
  fp.city_province,
  fp.categories,
  fp.academy_status,
  fp.quality_score,
  fp.status as fms_profile_status,
  fp.metadata as fms_metadata
from auth.users u
left join public.user_profiles up on up.auth_user_id = u.id
left join public.role_assignments ra
  on ra.user_profile_id = up.id
  and ra.role = 'fms'
left join public.fms_profiles fp on fp.user_profile_id = up.id
where lower(u.email) = lower('test-fms@example.com')
order by ra.created_at desc;
```

Expected result:

- `primary_role = fms`
- `user_profile_status = active`
- `assigned_role = fms`
- `role_assignment_status = active`
- `fms_profile_status = active`
- `fms_code` is present

## Table Editor Alternative

If you prefer not to run SQL:

1. In Authentication > Users, copy the Auth User ID for the FMS email.
2. In `user_profiles`, find or create the row where `auth_user_id` matches that Auth User ID.
3. Set `primary_role = fms`, `preferred_language = zh` or `en`, and `status = active`.
4. Copy `user_profiles.id`.
5. In `role_assignments`, find or create a row with:
   - `user_profile_id = user_profiles.id`
   - `role = fms`
   - `status = active`
6. In `fms_profiles`, find or create a row with:
   - `user_profile_id = user_profiles.id`
   - `fms_code = FMS-TEST-CN-001` or another unique code
   - `tier = silver`
   - `city_province = Guangzhou, Guangdong`
   - `categories = {"electronics","bags","household goods"}`
   - `academy_status = in_progress`
   - `quality_score = 85`
   - `status = active`
   - `metadata = {"languages":["zh-CN","en"],"repair_source":"manual_test_fms_account"}`

## After Repair

1. Login at `/fms/login`.
2. Open `/fms/dashboard`.
3. Login as admin.
4. Open `/admin/fms`.
5. Confirm the FMS appears as `Active / Assignable`.

If the test account was originally created through importer signup and should not remain an importer, revoke only the active importer role assignment after confirming no importer test flow still depends on it. Do not delete historical profile rows casually.
